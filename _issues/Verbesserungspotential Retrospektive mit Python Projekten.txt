Das Mapping funktioniert ueber einen Trampoline-Mechanismus in drei Phasen:

1. Mutanten-Erzeugung
Jede Funktion im Quellcode wird mit einem Trampoline umhuellt. Aus def add(a, b): return a + b wird intern:

# Original umbenannt
def x_add__mutmut_orig(a, b): return a + b

# Mutanten-Dict
x_add__mutmut_mutants = {
    'x_add__mutmut_1': lambda a, b: return a - b,   # + → -
    'x_add__mutmut_2': lambda a, b: return a * b,   # + → *
}

# Trampoline ersetzt die Original-Funktion
def add(*args, **kwargs):
    return _mutmut_trampoline(x_add__mutmut_orig, x_add__mutmut_mutants, *args, **kwargs)

2. Stats-Collection (Mapping aufbauen)
Vor dem eigentlichen Mutations-Lauf startet mutmut-win pytest als Subprocess mit MUTANT_UNDER_TEST=stats. Im Stats-Modus laesst das Trampoline die Originalfunktion laufen, merkt sich aber per record_trampoline_hit(): "Funktion X wurde aufgerufen."

3. Das Problem: Subprocess-Isolation
Und hier liegt die architektonische Limitierung: Der Stats-Lauf passiert in einem separaten Prozess. Die Trampoline-Hits landen im _state-Modul des Child-Prozesses — der Parent-Prozess sieht sie nie.

Aus runner.py:160-164:

"If trampoline hits can't be collected from subprocess (no shared state), fall back to mapping ALL tests to ALL mutants."

Deshalb die Warning no test-to-mutant mappings found und deshalb laeuft jeder Mutant gegen alle Tests — bei uns 52 Tests (kein Problem), beim anderen Projekt 914 Tests (massives Problem).

Fazit
Das Mapping-System ist implementiert aber funktional wirkungslos, weil die Subprocess-Architektur den Datentransfer verhindert. Um es zum Laufen zu bringen, muesste mutmut-win entweder die Hits in eine Datei/Datenbank schreiben oder pytest in-process ausfuehren (was aber laut Code-Kommentar wegen pytest-asyncio/hypothesis Haengern vermieden wird).
